# SEML E-commerce Recommendation ‚Äî Codebase Documentation

This document describes every component of the repository: what it is, what it
does, and how it connects to the rest of the system. It was generated by
scanning the full source tree on 2026-07-06.

## 1. What this project is

A CQRS-style, microservices e-commerce recommendation system built for
assignment AIMLCZG546 (Group 049). Given implicit user feedback (view, click,
cart, purchase events), it trains an **item-based collaborative-filtering**
model offline and serves top-k product recommendations online through a
read-only query API. A Streamlit dashboard is the presentation layer for both
the write (command) and read (query) sides.

## 3. Architecture

```text
Storefront / analyst
        ‚îÇ
        ‚ñº
  Streamlit UI (frontend/app.py)
        ‚îÇ
        ‚îú‚îÄ‚îÄ commands ‚îÄ‚îÄ‚ñ∫ Command Service (backend/src/ecom_ml/command_service)
        ‚îÇ                  ‚îú‚îÄ‚îÄ interaction event log (data/interactions.csv)
        ‚îÇ                  ‚îî‚îÄ‚îÄ ML training pipeline (backend/src/ecom_ml/ml)
        ‚îÇ                              ‚îÇ
        ‚îÇ                              ‚ñº
        ‚îÇ                    versioned model artifact (artifacts/)
        ‚îÇ                              ‚îÇ
        ‚îî‚îÄ‚îÄ queries ‚îÄ‚îÄ‚îÄ‚ñ∫ Query Service (backend/src/ecom_ml/query_service)
                                       ‚îÇ
                                       ‚ñº
                              top-k recommendations
```

- **Command Service** (port 8101) owns all writes: recording interaction
  events, creating user profiles, and triggering model training. Training
  produces a new, immutable, versioned artifact.
- **Query Service** (port 8102) is read-only. It loads the latest artifact and
  serves low-latency recommendation, catalogue, user, and recent-activity
  queries. It never touches the write path.
- **Streamlit UI** (port 8501, or 7860 in the combined container) calls both
  APIs over HTTP; it never reads data or model files directly, which keeps the
  CQRS service boundary honest.

The two services share the `ecom_ml` package (models, config, ML code) but run
as independent FastAPI processes with independent lifecycles.

## 4. Backend package ‚Äî `backend/src/ecom_ml/`

This is the installable Python package (see `pyproject.toml`,
`[tool.hatch.build.targets.wheel] packages = ["backend/src/ecom_ml"]`).

### 4.1 `config.py`

Defines `Settings`, a frozen dataclass holding the three filesystem paths the
services need: `data_path`, `users_path`, `artifact_dir`. `Settings.from_env()`
reads them from the environment variables `SEML_DATA_PATH`, `SEML_USERS_PATH`,
`SEML_ARTIFACT_DIR`, defaulting to `data/interactions.csv`, `data/users.csv`,
and `artifacts/` respectively. Both services call this once at app-creation
time; tests override the paths directly via `create_app(...)` keyword
arguments instead of environment variables.

### 4.2 `api_models.py`

Pydantic v2 models that form the typed contract shared by both services (and
consumed by the Streamlit UI's JSON responses):

- `Action` / `Interest` ‚Äî `Literal` type aliases enumerating the four allowed
  interaction actions (`view`, `click`, `cart`, `purchase`) and the five
  product categories used as cold-start interests.
- `InteractionCommand` ‚Äî write payload for `POST /commands/interactions`
  (`user_id`, `item_id`, `action`, optional `timestamp`).
- `TrainCommand` ‚Äî write payload for `POST /commands/train` (`k`,
  `holdout_per_user`, both bounded).
- `CreateUserCommand` ‚Äî write payload for `POST /commands/users` (`name`,
  `interest`).
- `UserProfileInfo` / `UserCommandResponse` / `UserCatalogResponse` ‚Äî read
  models for a single user, the response after creating one, and the full
  user list.
- `ProductInfo` / `RecommendationItem` / `RecommendationResponse` /
  `ProductCatalogResponse` ‚Äî read models for catalogue products and ranked
  recommendations.
- `RecentAction` / `RecentActionsResponse` ‚Äî read models for a user's most
  recent interaction history, enriched with product metadata.

### 4.3 `catalog.py`

A static, in-memory product catalogue used purely for human-readable display
(names/categories) ‚Äî it is **not** part of the trained model, which only
knows item IDs.

- `_PRODUCT_NAMES_BY_CATEGORY` ‚Äî a hard-coded dict of 12 product names for
  each of the 5 categories (60 products total).
- `Product` ‚Äî frozen dataclass (`item_id`, `name`, `category`).
- `_build_catalog()` / `PRODUCTS` / `PRODUCT_BY_ID` ‚Äî builds sequential IDs
  (`P001`‚Ä¶`P060`) at import time and indexes them by ID.
- `get_product(item_id)` ‚Äî looks up a product, falling back to a synthetic
  `Product(item_id, "Product {id}", "Other")` for any ID not in the static
  catalogue (keeps the API resilient to IDs generated outside this list).

### 4.4 `users.py`

Persistence and validation for **named user profiles** (as opposed to the
anonymous `u001`‚Ä¶`u080` synthetic users produced by `ml/data.py`). Backed by a
flat CSV file (`data/users.csv`).

- `UserProfile` ‚Äî frozen dataclass (`user_id`, `name`, `interest`,
  `created_at`) with `as_csv_row()`.
- `DEFAULT_USER_SPECS` / `generate_default_users(path)` ‚Äî writes 5
  deterministic demo profiles (Shreyas/Electronics, Sumanth/Home & Kitchen,
  Ravi/Fashion, Vivek/Personal Care, Nishant/Fitness & Lifestyle).
- `load_users(path)` ‚Äî reads and validates the CSV, raising if the file is
  missing or malformed.
- `create_user(name, interest)` ‚Äî builds a new profile with a slugified,
  collision-resistant ID (`usr-<slug>-<6 hex chars>`) and a UTC timestamp.
- `append_user(path, profile)` ‚Äî thread-safe (`_WRITE_LOCK`) append that
  rejects duplicate `user_id`s and validates before writing.
- `validate_user(profile)` ‚Äî shared validation: non-empty `user_id`/`name`,
  `interest` must be one of the 5 catalogue categories, `created_at` must be
  ISO-8601.

### 4.5 `command_service/main.py` ‚Äî the write side (port 8101)

`create_app(*, data_path=None, users_path=None, artifact_dir=None) -> FastAPI`
builds the FastAPI app; paths default to `Settings.from_env()` but can be
injected (used heavily by the test suite). A module-level `app = create_app()`
is the ASGI entrypoint uvicorn imports.

Endpoints:

| Method & path | Purpose |
|---|---|
| `GET /health` | Liveness probe; always returns `{"status": "ok", ...}`. |
| `POST /commands/interactions` | Accepts an `InteractionCommand`, builds an `Interaction` (server-generated `event_id`/`timestamp` if not supplied), and appends it to the CSV log via `append_interaction`. Returns HTTP 202. |
| `POST /commands/users` | Accepts a `CreateUserCommand`, creates and persists a `UserProfile`. Returns HTTP 201 with a `UserCommandResponse`. |
| `POST /commands/train` | Runs the full `train_pipeline` synchronously. Guarded by a `threading.Lock` (`training_lock`) so concurrent training requests get HTTP 409 instead of racing on the artifact files. |

All validation errors from the domain layer (`ValueError`) are translated to
HTTP 400; a busy training lock returns HTTP 409.

### 4.6 `query_service/main.py` ‚Äî the read side (port 8102)

`ModelRepository` is a small thread-safe cache: it stat()s
`artifacts/model_metadata.json`'s mtime and only calls `load_artifact()` again
when the file has changed, so the query service automatically picks up a
newly trained model without a restart, without reloading on every request.

`create_app(*, artifact_dir=None, data_path=None, users_path=None) -> FastAPI`
builds the app around one `ModelRepository` instance. `app = create_app()` is
the ASGI entrypoint.

Endpoints:

| Method & path | Purpose |
|---|---|
| `GET /health` | Returns `ok`/`model_version` if a model is loaded, `not_ready` otherwise. |
| `GET /queries/model-info` | Returns the full metadata JSON persisted alongside the model (version, timestamp, counts, evaluation metrics). 503 if no model exists yet. |
| `GET /queries/products` | Returns the full static catalogue (from `catalog.py`). |
| `GET /queries/users` | Returns all named user profiles (from `users.py`). |
| `GET /queries/recent-actions?user_id=&limit=` | Returns the user's `limit` most recent interaction events (newest first), enriched with product name/category. |
| `GET /queries/recommendations?user_id=&k=` | The core read endpoint ‚Äî see below. |

**Recommendation logic** (`recommendations` handler): if `user_id` exists in
the trained model's user index, it calls `model.recommend(user_id, k)`
(`strategy = "item-based-collaborative-filtering"`). If not ‚Äî e.g. a brand-new
named user with no interaction history yet ‚Äî it falls back to
**interest-based cold start**: it looks up the user's declared interest in
`users.csv` and returns the first `k` catalogue products in that category with
descending synthetic scores (`strategy = "interest-based-cold-start"`). An
unknown `user_id` that also isn't a registered profile returns HTTP 404.

The query service exposes **no write routes** ‚Äî posting a command against it
returns 404 (verified by `test_services.py`).

### 4.7 `ml/data.py` ‚Äî data loading and preprocessing

- `ACTION_WEIGHTS` ‚Äî implicit-feedback weighting: `view=1.0, click=2.0,
  cart=3.0, purchase=5.0`.
- `Interaction` ‚Äî frozen dataclass for one raw event (`event_id`, `timestamp`,
  `user_id`, `item_id`, `action`).
- `PreparedInteractions` ‚Äî frozen dataclass holding the validated,
  deduplicated, weighted **user √ó item matrix** plus the sorted `users`/`items`
  index tuples and raw/unique event counts.
- `generate_demo_interactions(path, user_count=80, item_count=60, seed=546)` ‚Äî
  deterministically synthesizes clustered shopping behavior: users are split
  into 5 clusters aligned with the 5 product-category blocks, each user gets
  10 "core" interactions from their cluster's items and 2 "neighbor"
  interactions from the next cluster, with actions sampled from a fixed
  probability distribution. This is what powers the reproducible demo dataset
  and is also called directly by tests with smaller `user_count`/`item_count`.
- `load_interactions(path)` ‚Äî reads and validates the CSV; raises if missing,
  malformed, or empty.
- `append_interaction(path, interaction)` ‚Äî thread-safe (`_WRITE_LOCK`)
  validated append, used by the command service.
- `validate_interaction(interaction)` ‚Äî required fields, action must be a
  known key in `ACTION_WEIGHTS`, timestamp must be ISO-8601.
- `prepare_interactions(rows)` ‚Äî deduplicates by `event_id` (last-write model
  is "first seen wins" via `dict.setdefault`), builds sorted user/item index
  maps, and accumulates weighted values into a dense `NDArray[np.float64]`
  matrix of shape `(len(users), len(items))`.

### 4.8 `ml/model.py` ‚Äî the collaborative-filtering model

- `Recommendation` ‚Äî frozen dataclass (`item_id`, `score`).
- `CollaborativeFilteringModel` ‚Äî frozen dataclass holding `users`, `items`,
  the raw `interaction_matrix`, the fitted `item_similarity` matrix, and a
  content-derived `version` string.
  - `.fit(prepared)` ‚Äî computes **item-item cosine similarity**
    (`sklearn.metrics.pairwise.cosine_similarity` on the transposed matrix),
    zeroes the diagonal (an item is never "similar to itself" for scoring
    purposes), and derives `version` as the first 12 hex characters of a
    SHA-256 hash over the matrix bytes plus the joined user/item ID lists ‚Äî
    i.e. the version changes if and only if the underlying data changes,
    giving reproducible, content-addressed model versioning.
  - `.recommend(user_id, k=5)` ‚Äî scores every item as
    `user_interaction_vector @ item_similarity`, masks out items the user has
    already interacted with (sets their score to `-inf`), sorts descending,
    and returns the top `k` as `Recommendation` objects with 4-decimal-rounded
    scores. Raises `KeyError` for an unknown `user_id` and `ValueError` if `k`
    is outside `[1, 20]`.

### 4.9 `ml/evaluation.py` ‚Äî leakage-safe offline evaluation

`evaluate_leave_n_out(prepared, k=5, holdout_per_user=2, seed=1546)`:

1. For each user with more than `holdout_per_user` positive interactions,
   randomly hides `holdout_per_user` of them (sets those matrix entries to 0)
   ‚Äî this is the **train** split; the hidden items are the **held-out
   ground truth**.
2. Fits a fresh `CollaborativeFilteringModel` on the train-only matrix (so the
   evaluation never sees the held-out signal ‚Äî no leakage).
3. For every evaluated user, generates top-`k` recommendations from the
   leakage-free model and compares them against that user's held-out set.

Returns `EvaluationMetrics` (frozen dataclass): `precision_at_k`,
`recall_at_k`, `hit_rate_at_k` (fraction of users with ‚â•1 correct hit),
`catalogue_coverage_at_k` (fraction of the catalogue that appears across all
users' top-k lists ‚Äî a diversity/reach measure), plus `evaluated_users`, `k`,
`holdout_per_user`. All rounded to 4 decimals; `.as_dict()` for JSON
serialization.

### 4.10 `ml/pipeline.py` ‚Äî orchestration

`train_pipeline(data_path, artifact_dir, k=5, holdout_per_user=2) ->
TrainingSummary` runs the full lifecycle end to end, printing a `[n/5]` stage
log to stdout as it goes:

1. **LOAD** ‚Äî `load_interactions`
2. **PREPARE** ‚Äî `prepare_interactions`
3. **EVALUATE** ‚Äî `evaluate_leave_n_out` (on the *original* data, using its own
   internal train/holdout split ‚Äî this is the offline quality gate)
4. **TRAIN** ‚Äî `CollaborativeFilteringModel.fit` on the *full* dataset (the
   model that actually gets served uses all available signal; only the
   evaluation step holds data back)
5. **PERSIST** ‚Äî `save_artifact`

Returns `TrainingSummary` (frozen dataclass): `model_type`, `version`,
`users`, `items`, `raw_events`, `unique_events`, `matrix_density`, `metrics`
(the evaluation dict), `model_path`. This is the object serialized as the
`summary` field of the command service's `POST /commands/train` response and
also printed by `scripts/train_and_evaluate.py`.

### 4.11 `ml/artifact.py` ‚Äî atomic model persistence

Persists the model **without pickle** (security/portability: only NumPy
arrays and JSON):

- `MODEL_FILENAME = "recommendation_model.npz"`,
  `METADATA_FILENAME = "model_metadata.json"`.
- `save_artifact(model, artifact_dir, metrics, training_event_count)` ‚Äî writes
  to temporary files (`.{name}.{uuid}`) first, then uses `os.replace` to
  atomically swap them into place. This means the query service (which polls
  the metadata file's mtime) never observes a half-written model. Metadata
  includes `model_type`, `version`, `trained_at` (UTC ISO-8601), `users`,
  `items`, `training_events`, and the nested `metrics` dict.
- `load_artifact(artifact_dir)` ‚Äî loads the `.npz` with `allow_pickle=False`,
  reconstructs the `CollaborativeFilteringModel`, loads the metadata JSON, and
  **cross-checks that `metadata["version"] == model.version`**, raising
  `ValueError` on mismatch (defends against a corrupted or partially-updated
  artifact pair).

## 5. Frontend ‚Äî `frontend/app.py`

A single-file Streamlit dashboard ("Recommendation Control Room") that talks
to both backend services purely over HTTP (`SEML_COMMAND_URL`,
`SEML_QUERY_URL` env vars, defaulting to `127.0.0.1:8101`/`8102`). No direct
file or model access ‚Äî this is what keeps the UI honestly decoupled from the
CQRS services.

Key pieces:

- `request_json(method, url, payload=, params=, timeout=)` ‚Äî a thin `httpx`
  wrapper that normalizes HTTP/JSON errors into a single `ApiError` exception
  the rest of the app can catch uniformly.
- `service_health(base_url)` ‚Äî calls `/health` with a short timeout and
  reduces the result to `(is_healthy, label, raw_json)` for the sidebar status
  cards.
- `inject_styles()` ‚Äî a large block of scoped CSS (custom properties, hero
  banner, status cards, recommendation/activity cards, form styling) injected
  via `st.markdown(..., unsafe_allow_html=True)`. Purely presentational.
- `render_status_card`, `render_recommendations`, `render_recent_actions`,
  `render_model_snapshot` ‚Äî presentation-only render functions for the
  sidebar, the ranked product grid, the "last 3 actions" strip, and the model
  metadata/metrics panel.
- Four tabs, each backed by a render function:
  - **`render_recommend_tab`** ‚Äî CQRS *query*: picks a user, calls
    `GET /queries/recommendations` then `GET /queries/recent-actions`, and
    caches both results in `st.session_state` so they survive reruns.
  - **`render_interaction_tab`** ‚Äî CQRS *command*: posts one interaction event
    to `POST /commands/interactions`. Explicitly tells the user that
    retraining is required before the new event affects recommendations
    (making the CQRS write/read lag visible rather than hidden).
  - **`render_users_tab`** ‚Äî lists existing named users
    (`GET /queries/users`) and posts new ones (`POST /commands/users`).
  - **`render_training_tab`** ‚Äî triggers `POST /commands/train` with
    user-chosen `k`/holdout sliders and displays the returned summary as JSON.
- `main()` ‚Äî page config, health checks, prefetches model info/products/users,
  renders the sidebar and hero banner (showing the currently loaded model
  version), then lays out the four tabs plus the model snapshot footer.

## 6. Scripts ‚Äî `scripts/`

CLI entry points, all runnable as `python scripts/<name>.py` and mirrored as
`make` targets.

- **`seed_data.py`** ‚Äî `generate_demo_interactions` + `generate_default_users`
  into `data/interactions.csv` and `data/users.csv` (paths overridable via
  `--out`/`--users`). This is the "reset to a known reproducible dataset" step.
- **`train_and_evaluate.py`** ‚Äî thin CLI over `train_pipeline`; prints the
  `TrainingSummary` as JSON. Args: `--data`, `--artifacts`.
- **`run_demo.py`** ‚Äî exercises **already-running** services over real HTTP:
  posts one interaction, triggers training, queries recommendations and
  model-info, and prints/optionally saves (`--save`) the full transcript. Its
  `execute_demo(command_url, query_url)` function is imported directly by
  `verify_live.py`.
- **`verify_live.py`** ‚Äî the one-command end-to-end check. It regenerates
  data, trains a model, **starts both services as subprocesses** on
  `127.0.0.1:8101`/`8102` with logs redirected to `evidence/*.log`, waits for
  `/health` via `wait_for()`, runs `execute_demo`, asserts exactly 5
  recommendations came back, writes `evidence/live_demo.json`, and always
  terminates (`finally`) both subprocesses cleanly (`terminate()` then
  `kill()` after a 5s grace period). This is the script CI-style checks and
  the report/notebook generators depend on for real execution evidence.

Generators for the assignment's submission artifacts (not part of the running
application).

- **`generate_assets.py`** ‚Äî builds hand-drawn-style GR4ML diagrams with raw
  `matplotlib` patches (rectangles, ellipses, hexagons for "algorithm", cloud

  outputs. Writes `final_submission/{GROUP}.ipynb`.

## 8. Tests ‚Äî `backend/tests/`

Run via `pytest` (configured in `pyproject.toml`: `testpaths =
["backend/tests"]`, `pythonpath = ["backend/src"]`).

- **`test_ml_pipeline.py`** ‚Äî unit tests for the ML lifecycle in isolation
  (using `tmp_path` fixtures, no HTTP):
  - loading/preprocessing/deduplication counts and matrix shape;
  - invalid actions are rejected with a matching error message;
  - `recommend()` never returns items the user has already interacted with,
    and results are score-sorted descending;
  - leave-N-out metrics are always within `[0, 1]`;
  - a trained pipeline's artifact reloads to an equivalent, working model
    (`summary.version == reloaded.version == metadata["version"]`).
- **`test_services.py`** ‚Äî integration tests using FastAPI's `TestClient`
  against `create_command_app`/`create_query_app` with injected temp paths:
  - the command service accepts a write, trains, creates a user, and persists
    all of it;
  - the query service serves recommendations (both trained-model and
    interest-based cold-start strategies), the product catalogue, the user
    list, and recent actions ‚Äî and **rejects write attempts with 404**
    (proving the CQRS read/write separation at the HTTP boundary);
  - an unknown, unregistered `user_id` returns 404 from `/queries/recommendations`.

## 9. Deployment & local orchestration

### 9.1 `Dockerfile`

Single image, `python:3.11-slim` base. Copies the whole repo, installs
`requirements.txt` then the package itself (`pip install -e .`), makes
`start.sh` executable, exposes port `7860` (the Hugging Face Spaces
convention ‚Äî see the `sdk: docker` / `app_port: 7860` front-matter in
`README.md`), and runs `start.sh` as the container command. One image serves
all three roles (command/query/ui) depending on which command overrides it at
runtime.

### 9.2 `docker-compose.yml`

Three services built from the same image/context, each overriding `command`:

- `command-service` ‚Äî port 8101, read-write mounts for `./data` and
  `./artifacts`, `SEML_*` env vars pointing at `/app/...`, a Python-`urllib`
  based healthcheck against `/health`.
- `query-service` ‚Äî port 8102, **read-only** mounts (`:ro`) for `data` and
  `artifacts` ‚Äî enforced at the container level, not just by convention ‚Äî
  `depends_on: command-service (service_healthy)`.
- `ui` ‚Äî port 8501, depends on both services being healthy, points
  `SEML_COMMAND_URL`/`SEML_QUERY_URL` at the other containers' Compose DNS
  names.

### 9.3 `start.sh`

The all-in-one entrypoint used by the Docker image (and Hugging Face Spaces):
starts the command service on `127.0.0.1:8101` and query service on
`127.0.0.1:8102` in the background, waits 3 seconds, verifies both PIDs are
still alive (fails fast with a clear message if either crashed on boot), then
runs Streamlit in the foreground on the public port `7860` with
`--server.headless=true`.

### 9.4 `Makefile`

Convenience wrapper around every workflow command: `install`, `seed`,
`train`, `command`, `query`, `ui`, `demo`, `verify`, `lint`, `format`,
`typecheck`, `test`, `cov`, `check` (the full local CI gate: lint + typecheck
+ cov + format-check), `notebook`, `word`, `report`, `docker-build`, `up`,
`down`, `logs`, `clean`. `make help` prints all targets with their doc
comments.

## 11. Configuration files

- **`pyproject.toml`** ‚Äî Hatchling build backend; runtime deps pinned exactly
  (`fastapi`, `httpx`, `numpy`, `pydantic`, `scikit-learn`, `streamlit`,
  `uvicorn[standard]`); optional `dev` (mypy, pre-commit, pytest(-cov), ruff)
  and `report` (ipykernel, matplotlib, nbclient, nbformat, pillow,
  python-docx, reportlab) extras. Ruff targets `py311`, line length 100, with
  a specific lint rule selection (`E, W, F, I, UP, B, C4, SIM, RET, RUF`).
  Mypy runs in strict-ish mode (`disallow_untyped_defs`,
  `disallow_incomplete_defs`, `no_implicit_optional`) over
  `backend/src`, `frontend`, `scripts`, with `ignore_missing_imports` carved
  out only for `sklearn`, `matplotlib`, `docx`, `PIL`.
- **`requirements.txt`** ‚Äî the same exact-pinned runtime dependency list, used
  directly by the Dockerfile and CI (kept in sync with the `[project]`
  dependencies in `pyproject.toml`).
- **`.env.example`** ‚Äî documents the six environment variables the services
  and scripts read: `SEML_DATA_PATH`, `SEML_USERS_PATH`, `SEML_ARTIFACT_DIR`,
  `SEML_COMMAND_URL`, `SEML_QUERY_URL`.
- **`.pre-commit-config.yaml`** ‚Äî local git hook wiring for the same
  lint/format/type checks CI runs (keeps issues out of commits, not just out
  of CI).
- **`.streamlit/config.toml`** ‚Äî Streamlit server/theme configuration for the
  dashboard.
- **`submission_details.json`** ‚Äî assignment metadata (group number, course,
  deadline, per-member contribution percentages) consumed by
  `tools/build_report.py` and `tools/build_notebook.py`. Contains `TO_FILL`
  placeholders that must be replaced before final submission.

## 12. Data and artifacts

- **`data/interactions.csv`** ‚Äî the raw interaction event log
  (`event_id, timestamp, user_id, item_id, action`), the append target for
  both `generate_demo_interactions` and the command service's
  `/commands/interactions` endpoint.
- **`data/users.csv`** ‚Äî named user profiles (`user_id, name, interest,
  created_at`), the append target for `generate_default_users` and
  `/commands/users`.
- **`artifacts/recommendation_model.npz`** ‚Äî the persisted model: `users`,
  `items`, `interaction_matrix`, `item_similarity`, `version` arrays, written
  by `save_artifact`.
- **`artifacts/model_metadata.json`** ‚Äî human-readable sibling of the above:
  version, training timestamp, counts, and the full evaluation metrics dict;
  the query service uses this file's mtime to detect a fresh model.
