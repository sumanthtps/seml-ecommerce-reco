4. Objective 1 - Requirements Formulation

4.1 Domain and Problem Statement
Write the problem as a business problem first and an ML problem second. The business problem is
revenue and engagement. The ML problem is ranking products for each user using interaction history.
Suggested report text
An e-commerce platform wants to increase revenue per shopping session by showing each shopper
products they are likely to engage with or buy. User behaviour such as product views, clicks, add-to-cart
actions, and purchases arrives continuously from the storefront. The system must convert this behaviour
into a personalised top-N product list in near real time. Therefore, the application is framed as a
recommendation/ranking problem: from the evolving user-item interaction history, rank catalogue items for
a given user and serve the highest scoring products through the storefront.

5. GR4ML Views - What to Draw and Explain
GR4ML diagrams should not be decorative. They show how business strategy leads to an analytics
solution and then to data preparation. In your report, place each diagram first and then explain the
diagram in 5-7 sentences.
5.1 Business View
 Figure 1: Business View for the selected e-commerce recommendation scenario.
How to explain it: The strategic goal is to increase revenue per session through cross-sell. This goal is
measured by business indicators such as click-through rate and average order value. The business
decision is which products to show to a shopper. The decision is informed by the question: what items
will this user most likely engage with? The insight that answers the question is a personalised top-N
recommendation list delivered in real time.
5.2 Analytics Design View

Page 7
 Figure 2: Analytics Design View showing the prediction goal and algorithm trade-offs.
How to explain it: The insight from the Business View becomes a prediction analytics goal: rank items
by user preference. Candidate algorithms include item-based collaborative filtering, matrix
factorisation, and content-based filtering. Low latency and recommendation quality are the main
softgoals. Item-based collaborative filtering is selected for the real-time path because item similarity
can be precomputed and serving can be fast. Matrix factorisation may improve quality but has higher
complexity, so it can be described as a future enhancement.
5.3 Data Preparation View
 Figure 3: Data Preparation View from raw events to prepared ranking features.
How to explain it: The data sources are user activity events, orders, and product catalogue metadata.
Preparation tasks include deduplicating repeated events, converting actions into numeric weights,
building a user-item interaction matrix, computing item-item similarity, and filtering already-seen
products. The prepared outputs are the interaction matrix and item-similarity table, which feed the
item ranking goal.

6.2 Data Flow
• The shopper browses products on the storefront. Product views, clicks, cart actions, and purchases
are sent as activity events.
• All external calls first hit the API Gateway. The gateway authenticates, rate-limits, and routes
requests to back-end services.
• Activity events are routed to the Event Tracking Service, which validates the event and publishes it to
the broker/queue.
• The Event Consumer reads events asynchronously, applies action weights, and updates the user-item
interaction matrix in the Feature Store.
• The Recommendation Service reads the interaction matrix and item-similarity table, computes item
scores, filters already-seen items, and returns top-N products.
• The gateway returns the top-N JSON response to the storefront. The cache may be used to reduce
repeated computation.

6.3 Architectural Pattern 1 - Event-Driven Architecture
In Event-Driven Architecture, producers publish events and consumers react to those events
asynchronously. In this assignment, the storefront/event tracking service is the producer, the
broker/queue is the event transport, and the consumer updates recommendation features. This pattern
is suitable because user behaviour is continuous and bursty. The recommendation serving path should
not wait for every event to be processed synchronously.
What to show in implementation: a producer endpoint that accepts user events, a queue/broker where
events are stored, and a consumer worker that reads events and updates the feature store.
6.4 Architectural Pattern 2 - API Gateway
API Gateway provides one stable external API for clients. Instead of the storefront calling many internal
services directly, it calls the gateway. The gateway performs cross-cutting concerns such as
authentication, rate limiting, logging, and routing. In this scenario, GET /recommend?user=u7 should
be exposed through the gateway and internally routed to the Recommendation Service.
What to show in implementation: a gateway service with a /recommend endpoint that checks a token or
demo header, applies a simple rate-limit rule, calls the recommendation service, and returns its JSON
response.

7.4 Core Idea of the Recommendation Logic
Each event is converted into a numeric weight. A view has smaller weight, a purchase has higher
weight. The feature store maintains a user-item matrix M where each row is a user and each column is
a product. Item similarity is computed from the transpose of this matrix. For a user u, recommendation
score is calculated as user interactions multiplied by item similarity. Already-seen products are
removed from the output.
Step
Formula / rule
Meaning

Stronger actions indicate stronger preference.

Item similarity
sim = cosine_similarity(M.T)
Items are similar if users interact with them in
similar patterns.
CF scoring
scores = M[user] @ sim
Score unseen items based on similarity to items the
user interacted with.

8.1 feature_store.py
import numpy as np
from sklearn.metrics.pairwise import cosine_similarity
USERS = [f"u{i}" for i in range(1, 11)]
ITEMS = [f"P{i:02d}" for i in range(1, 21)]
USER_INDEX = {u: i for i, u in enumerate(USERS)}
ITEM_INDEX = {p: i for i, p in enumerate(ITEMS)}
WEIGHT = {"view": 1, "click": 2, "cart": 3, "purchase": 5}
MATRIX = np.zeros((len(USERS), len(ITEMS)), dtype=float)
ITEM_SIMILARITY = np.zeros((len(ITEMS), len(ITEMS)), dtype=float)
EVENT_COUNT = 0
def increment(user_id: str, item_id: str, action: str) -> None:
    global EVENT_COUNT
    if user_id not in USER_INDEX or item_id not in ITEM_INDEX:
        raise ValueError("unknown user or item")
    MATRIX[USER_INDEX[user_id], ITEM_INDEX[item_id]] += WEIGHT.get(action, 1)
    EVENT_COUNT += 1
def refresh_similarity() -> None:
    global ITEM_SIMILARITY
    ITEM_SIMILARITY = cosine_similarity(MATRIX.T)
    np.fill_diagonal(ITEM_SIMILARITY, 0.0)
def rank_items(user_id: str, k: int = 5) -> list[dict]:
    if user_id not in USER_INDEX:
        raise ValueError("unknown user")
    u = USER_INDEX[user_id]
    scores = MATRIX[u] @ ITEM_SIMILARITY
    scores[MATRIX[u] > 0] = -np.inf
    top = np.argsort(scores)[::-1][:k]
    return [{"item": ITEMS[i], "score": round(float(scores[i]), 3)} for i in top]
def seed_demo_data() -> None:
    events = [
        ("u7", "P01", "view"), ("u7", "P02", "click"),
        ("u7", "P05", "cart"), ("u7", "P09", "purchase"),
        ("u1", "P01", "purchase"), ("u1", "P14", "cart"),
        ("u2", "P02", "purchase"), ("u2", "P14", "click"),
        ("u3", "P05", "purchase"), ("u3", "P17", "cart"),
        ("u4", "P09", "purchase"), ("u4", "P11", "click"),
        ("u5", "P01", "cart"), ("u5", "P17", "purchase"),
    ]
    for user, item, action in events:
        increment(user, item, action)
    refresh_similarity()
8.2 event_tracking_service.py
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
from queue import Queue
app = FastAPI(title="Event Tracking Service")
EVENT_QUEUE = Queue()
class ActivityEvent(BaseModel):
    user_id: str
    item_id: str
    action: str
@app.post("/track")
def track(event: ActivityEvent):
    if event.action not in {"view", "click", "cart", "purchase"}:
        raise HTTPException(status_code=400, detail="invalid action")
    EVENT_QUEUE.put(event.model_dump())
    return {"status": "accepted", "queued_event": event.model_dump()}
@app.get("/queue-size")
def queue_size():
    return {"pending_events": EVENT_QUEUE.qsize()}

8.3 combined_event_app.py - easiest local Event-Driven Demo
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
from queue import Queue
from threading import Thread
import time
import feature_store
app = FastAPI(title="Event Driven Recommendation Updater")
EVENT_QUEUE = Queue()
feature_store.seed_demo_data()
class ActivityEvent(BaseModel):
    user_id: str
    item_id: str
    action: str
def consumer_loop():
    while True:
        event = EVENT_QUEUE.get()
        try:
            feature_store.increment(event["user_id"], event["item_id"], event["action"])
            feature_store.refresh_similarity()
            print("processed event:", event)
        except Exception as exc:
            print("failed event:", event, exc)
        time.sleep(0.1)
@app.on_event("startup")
def start_consumer():
    Thread(target=consumer_loop, daemon=True).start()
@app.post("/track")
def track(event: ActivityEvent):
    if event.action not in {"view", "click", "cart", "purchase"}:
        raise HTTPException(status_code=400, detail="invalid action")
    EVENT_QUEUE.put(event.model_dump())
    return {"status": "accepted", "pattern": "event-driven", "event": event.model_dump()}
@app.get("/rank")
def rank(user_id: str, k: int = 5):
    return {
        "user_id": user_id,
        "strategy": "item-based-collaborative-filtering",
        "recommendations": feature_store.rank_items(user_id, k)
    }

from fastapi import FastAPI, Header, HTTPException
import httpx
import time
app = FastAPI(title="API Gateway")
RECO_SERVICE = "http://127.0.0.1:8001"
REQUEST_LOG = {}
VALID_TOKEN = "Bearer seml-demo-token"
def valid_token(authorization: str | None) -> bool:
    return authorization == VALID_TOKEN
def rate_limited(user_id: str) -> bool:
    now = time.time()
    last = REQUEST_LOG.get(user_id, 0)
    REQUEST_LOG[user_id] = now
    return now - last < 0.5
@app.get("/recommend")
async def recommend(user_id: str, k: int = 5, authorization: str | None = Header(None)):
    if not valid_token(authorization):
        raise HTTPException(status_code=401, detail="unauthorized")
    if rate_limited(user_id):
        raise HTTPException(status_code=429, detail="too many requests")
    async with httpx.AsyncClient() as client:
        response = await client.get(f"{RECO_SERVICE}/rank", params={"user_id": user_id, "k":
            k})
    return response.json()

• Do not submit only an implementation notebook without the report explanation.
• Do not forget screenshots; the submission guideline explicitly requires application screenshots with
explanation.
• Do not draw GR4ML diagrams without explaining the meaning of each element.
• Do not mention Event-Driven Architecture and API Gateway only in theory; show where each pattern
appears in the code and architecture.

12. Viva / Review Questions with Answers
Question
Answer
Why is this an ML-based application?
The system learns user preference patterns from historical and streaming
user-item interactions and uses those patterns to rank products.
Why choose item-based collaborative
filtering?
It is simple, explainable, and fast for serving because item similarity can be
precomputed and reused.
Why event-driven architecture?
User activity arrives continuously and at high volume. Events allow ingestion
and feature updates to happen asynchronously without blocking
recommendation requests.
Why API Gateway?
It gives clients one secured endpoint and centralises routing, authentication,
and rate limiting.
Which components are ML components?
The recommendation service is the main ML component. Feature store
supports ML serving by storing interaction matrix and item similarity.
Which components are non-ML?
API Gateway, event tracking service, message broker/queue, and event
consumer are software/infrastructure components.
How do you measure success?
Precision@5, CTR uplift, p95 latency, event-to-feature lag, and catalogue
coverage.
What is a limitation of the prototype?
It uses a simplified local queue and small in-memory data. In production it
should use Kafka/Redis, persistent feature store, monitoring, and A/B testing.
